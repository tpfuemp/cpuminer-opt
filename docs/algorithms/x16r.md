# x16r and the X16 family

**Coins:** Genix (GENIX), and others across the variants below. (Ravencoin
introduced x16r but has since moved to the KawPow algorithm.)
**Algorithm names:** `x16r`, `x16rv2`, `x16rt`, `x16rt-veil`, `x16s`, `hex`, `x20r`, `x21s`
**Family:** header-selected chain of core hashes (no memory-hard rounds)

```
./cpuminer -a x16r -o stratum+tcp://<pool>:<port> -u <wallet> -p x
```

---

## Overview

x16r runs a chain of 16 well-known cryptographic hashes, where the **order** of the
hashes is chosen by the block header. Because the order changes every block, mining
hardware cannot pipeline a fixed sequence — the design's goal of ASIC resistance.
Unlike [GhostRider](ghostrider.md) and [Flex](flex.md), there are no CryptoNight
(memory-hard) rounds, so x16r is compute-bound and parallelizes extremely well.

The 32-byte result of the chain is the proof-of-work hash compared against the
target.

## Structure

1. **The 16 hashes.** All 512-bit:
   `blake, bmw, groestl, jh, keccak, skein, luffa, cubehash, shavite, simd, echo,
   hamsi, fugue, shabal, whirlpool, sha512`.

2. **Order selection.** The order is read from the previous-block-hash field of the
   header. The 16 hex nibbles of its last 8 bytes form a 16-character order string;
   each nibble (`0`–`f`) names the hash to run at that position. In `x16r`, **hashes
   may repeat and some may not appear at all** — the order is effectively random per
   block.

3. **The chain.** The 16 rounds run in the selected order, each consuming the
   previous round's 64-byte output (the first round consumes the 80-byte header), the
   final round's output being the PoW hash.

## Family variants

All share the 16-hash core and differ only in how the order is derived (or in extra
rounds):

| Algo | How it differs |
|---|---|
| `x16r` | Order from the previous block hash; repeats allowed. |
| `x16s` | Order is a **permutation** (each hash exactly once), shuffled deterministically from the previous block hash. |
| `x16rt` | Order from a **time-based** hash (derived from the block timestamp) instead of the previous block hash. |
| `x16rt-veil` | `x16rt` ordering with Veil's header handling. |
| `x16rv2` | `x16r` plus an additional injected hash step, to further disrupt fixed-pipeline hardware. |
| `hex` | An `x16r`-based ordering variant. |
| `x20r` | Extends the chain with additional hashes. |
| `x21s` | Extends `x16s` with several extra finalizing hashes. |

## Performance

x16r is among the most parallel-friendly algorithms in the project, because it has no
memory-hard step:

- **Parallel-lane hashing.** It ships 8-way (AVX-512), 4-way (AVX2), 2-way, and a
  scalar fallback — 8 independent nonces hashed at once on AVX-512 CPUs.
- **Hardware AES.** The AES-based rounds (Groestl, Echo, Shavite, Fugue) use AES-NI
  where available.
- The hash order depends only on the header's prev-block (or time) field, not the
  nonce, so it is computed once per block and reused across all nonces in a lane
  group.

## Verification

Each implementation (every SIMD width and the scalar reference) produces byte-for-byte
identical output; correctness is confirmed by pool-accepted shares.

## Possible optimizations (preview)

x16r is already close to the practical ceiling for a core-hash chain; remaining
candidates are incremental:

- **VAES rounds** — vectorized AES for the AES-based hashes on supporting CPUs.
- **Wider coverage** — ensure every core hash has an AVX-512 lane implementation so
  the 8-way path never falls back to a narrower width mid-chain.
- **Final-round early exit** — skip remaining work for a nonce once an intermediate
  result cannot possibly meet the target.
